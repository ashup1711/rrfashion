# Comprehensive Review of R R Fashion — Project Plan

This is a well-structured, thorough plan. Below is a detailed review covering schema gaps, architectural concerns, missing modules, timeline realism, and concrete recommendations — each point is intended to be directly actionable.

---

## 1. Database Schema — Critical Gaps

The schema in §6 is a useful starting point but has several missing tables and structural issues that will cause rework if not addressed before Phase 1.

### 1.1 Missing Tables

```
-- Referenced but not defined
categories (id, name, parent_id, ...)
store_locations (id, name, address, state, gstin, phone, is_active, ...)
product_images (id, variant_id, url, alt_text, sort_order, is_primary, size_type)

-- Absent entirely, but required by stated requirements
rental_bookings (
  id, order_item_id, unit_id, status,
  booked_at, pickup_at, due_return_at, actual_return_at,
  deposit_amount, deposit_status, late_fee, damage_charge,
  inspected_by_admin_id, inspected_at, notes
)
-- Separating rental lifecycle from order_items is essential.
-- order_items captures the financial line; rental_bookings captures
-- the operational lifecycle (pickup → return → inspection → close).

inventory_locks (
  id, unit_id, store_id, locked_by_admin_id, device_id,
  locked_at, expires_at, reason, status[active/released/expired],
  client_uuid
)
-- Locks are ephemeral. Embedding locked_until in inventory_units
-- means every lock create/release is a write to your main inventory
-- table. A separate table lets you query active locks efficiently,
-- audit lock history, and reconcile without touching the units table.

coupons (
  id, code, type[percent/flat], value, min_cart_value,
  max_discount, applies_to[sale/rent/all],
  category_ids, brand_ids,
  usage_limit, used_count,
  valid_from, valid_until, is_active
)
coupon_usages (id, coupon_id, user_id, order_id, created_at)

notifications (
  id, user_id, type, channel[email/sms/push],
  title, body, data_json, status[pending/sent/failed],
  sent_at, created_at
)

product_tags (id, product_id, key, value)
-- For flexible attribute filtering (fabric, occasion, season)
-- without altering the products schema for every new attribute.

size_guide (id, category_id, measurements_json, image_url)

otp_verifications (id, phone, otp_hash, purpose, verified, expires_at, attempts)
```

### 1.2 Schema Structural Issues

**`inventory_units` vs aggregate stock:**
The current model tracks individual units (necessary for rentals — each unit has its own lifecycle). But for sale-only items, tracking individual rows per unit is expensive when you have 500 identical t-shirts. Recommend a hybrid:

```
inventory_units — used for rental-eligible items (individual tracking)
inventory_summary (variant_id, store_id, channel, quantity_available, quantity_reserved, quantity_sold)
  — used for sale-only items (aggregate count)
```

At checkout, the system checks which model applies. For rental items, specific `unit_id` is reserved. For sale-only items, `quantity_available` is decremented atomically. This avoids creating 500 rows for 500 identical sale-only t-shirts while still tracking individual rental pieces.

**`payments` table needs offline payment recording:**
```
payments (
  ...,
  channel[online/offline],
  method[card/upi/netbanking/wallet/cash/pos_card],
  offline_received_by_admin_id,    -- who accepted cash/card
  offline_reference,               -- card terminal ref, UPI txn ID noted manually
)
```

**`invoices` — sequential numbering safety:**
Gapless sequential numbering in a concurrent system requires a Postgres function with advisory locks. A simple `MAX()+1` will race:

```sql
CREATE OR REPLACE FUNCTION next_invoice_number(p_store_id INT, p_fin_year TEXT)
RETURNS TEXT AS $$
DECLARE
  next_seq INT;
  lock_key BIGINT := (p_store_id::BIGINT * 100000 + p_fin_year::BIGINT);
BEGIN
  PERFORM pg_advisory_xact_lock(lock_key);
  INSERT INTO invoice_sequences (store_id, financial_year, last_number)
  VALUES (p_store_id, p_fin_year, 1)
  ON CONFLICT (store_id, financial_year)
  DO UPDATE SET last_number = invoice_sequences.last_number + 1
  RETURNING last_number INTO next_seq;
  RETURN p_fin_year || '/' || p_store_id || '/' || LPAD(next_seq::TEXT, 6, '0');
END;
$$ LANGUAGE plpgsql;
```

**E-invoicing consideration (India):** If R R Fashion's aggregate turnover exceeds ₹5 Cr (current threshold), generating an IRN via the NIC e-invoice API is mandatory. Even if below threshold now, building the hook early is wise:

```
invoices (..., e_invoice_irn, e_invoice_ack_date, e_invoice_status)
```

## 2. Architecture — Contradictions & Gaps

### 2.1 Diagram vs. Recommendation Contradiction

§2 shows a microservices diagram but the text recommends modular monolith. This is confusing for any developer onboarding. **Fix the diagram** to show the modular monolith, or at minimum add a clear note:

> "The diagram illustrates logical module boundaries. In Phase 1, these are modules within a single Node.js process communicating via function calls, not network calls. They share one PostgreSQL database. Future split into separate services is supported by the module boundaries but not implemented initially."

### 2.2 Missing: API Versioning Strategy

No mention of how the API evolves. Recommend:

```
/api/v1/products
/api/v1/orders
```

With a versioning policy: vN is supported for at least 12 months after vN+1 launches. Breaking changes never land in the same version.

### 2.3 Missing: API Error Contract

Every API response should follow a consistent envelope:

```json
{
  "success": true,
  "data": { ... },
  "meta": { "page": 1, "total": 47 }
}
```

```json
{
  "success": false,
  "error": {
    "code": "STOCK_INSUFFICIENT",
    "message": "Only 2 units available for SKU-RED-M",
    "details": [{ "field": "qty", "issue": "Requested 3, available 2" }]
  }
}
```

This is critical for the POS sync engine — conflict errors need structured, machine-parseable codes, not human strings.

### 2.4 Missing: Database Migration Strategy

No mention of how schema changes are managed. Recommend:

- **Knex.js migrations** (if using Express/Fastify) or **TypeORM/Prisma migrations** (if using NestJS)
- Migrations run as a Kubernetes Job on deploy, before the app pods start
- Every migration must be reversible (down method)
- Seed scripts separate from migrations

### 2.5 Missing: Observability Specifics

The plan mentions Prometheus + Grafana and ELK/Loki but doesn't specify what to actually instrument:

```
Metrics to track from day one:
  - http_request_duration_seconds (by route, method, status)
  - http_requests_total (by route, method, status)
  - db_query_duration_seconds (by query type)
  - redis_command_duration_seconds
  - order_created_total (by channel, type)
  - payment_processed_total (by method, status)
  - inventory_conflict_total (critical for offline sync)
  - pos_sync_lag_seconds (time between offline action and server confirmation)
  - pos_pending_outbox_count (per device)
  - rental_overdue_total
```

The `inventory_conflict_total` and `pos_sync_lag_seconds` metrics are uniquely important to this system — they tell you whether the offline-first architecture is actually working in production.

## 3. Rental Module — Deeper Design Needed

The plan covers rental requirements at a feature level but the operational model needs more precision. Rental is the most complex part of this system — more complex than standard e-commerce — and underspecifying it leads to the most painful rework.

### 3.1 Rental State Machine

```
                    ┌──────────┐
                    │  booked  │
                    └────┬─────┘
                         │ (pickup confirmed)
                    ┌────▼─────┐
              ┌─────│ in_use   │─────┐
              │     └────┬─────┘     │
         (late return)  │(returned)  │(reported lost)
              │    ┌────▼──────┐     │
              │    │ returned  │     │
              │    └────┬──────┘     │
              │         │            │
              │    ┌────▼──────┐     │
              │    │inspected  │     │
              │    └──┬───┬────┘     │
              │       │   │          │
              │  (ok) │   │(damaged) │
              │       │   │          │
              │  ┌────▼──┐│  ┌──────▼───────┐
              │  │closed ││  │damage_assessed│
              │  └───────┘│  └──────┬───────┘
              │           │         │
              │           │    ┌────▼──────┐
              │           │    │repair/wash│
              │           │    └────┬──────┘
              │           │         │(ready)
              │           │    ┌────▼──────┐
              │           │    │ available │
              │           │    └───────────┘
         ┌────▼──────┐    │
         │late_return│────┘
         └───────────┘  (then → returned → inspected)
```

Each transition should have:
- Who can trigger it (role)
- What data is required (damage notes, photos, deposit adjustment)
- What side effects fire (notification, deposit refund, inventory status update, late fee invoice)

### 3.2 Rental Availability Query

This is the hardest query in the system. For a given variant between `rent_start` and `rent_end`:

```sql
-- Count units that are NOT booked for overlapping dates
SELECT COUNT(*) AS available
FROM inventory_units iu
WHERE iu.variant_id = :variant_id
  AND iu.store_id = :store_id
  AND iu.status IN ('available', 'returned')  -- rentable statuses
  AND iu.id NOT IN (
    SELECT rb.unit_id
    FROM rental_bookings rb
    WHERE rb.status NOT IN ('closed', 'cancelled')
      AND rb.booked_at IS NOT NULL
      AND (rb.pickup_at <= :rent_end AND rb.due_return_at >= :rent_start)
  );
```

At scale, this needs proper indexing on `rental_bookings(unit_id, status, pickup_at, due_return_at)` and potentially a dedicated `unit_calendar` table or a `daterange` column with GiST index for efficient overlap queries.

### 3.3 Rental Extension

Not addressed in the plan. A customer wants to extend their rental mid-rental. This is a common scenario. Need:

```
rental_extensions (
  id, rental_booking_id, original_due_date, new_due_date,
  additional_charge, payment_id, approved_by_admin_id, created_at
)
```

The system must re-check availability for the extended period before allowing it.

### 3.4 Security Deposit Accounting

The plan mentions Razorpay pre-auth and offline cash deposits but doesn't detail the accounting side. For Indian GST compliance:

- Security deposit is **not** a sale — no GST is levied on it at collection
- When deposit is partially forfeited (damage/late fee), the forfeited amount **becomes** a taxable supply (late fee = service, damage charge = supply)
- This means the credit note / supplementary invoice for deposit forfeiture must include GST

The `payments` and `invoices` tables need to distinguish:
```
payments.type IN ('sale_payment', 'rental_charge', 'security_deposit', 'deposit_refund', 'late_fee', 'damage_charge')
```

And the invoice generation logic must create separate line items for each taxable component.

## 4. Offline-First POS — Additional Edge Cases

The §8 architecture is solid. Below are scenarios not covered that will bite in production:

### 4.1 Price Changes While Device Is Offline

If admin updates a product's price while a POS device is offline, the device will sell at the old price. On sync, the order is already completed.

**Recommendation:** On sync, the server checks if the price used in the offline order matches the current price. If it doesn't match:
- If the price difference favors the store (customer paid less), flag the order for admin review — don't auto-adjust a completed transaction
- Log the discrepancy in `audit_logs`
- The background sync pulling latest prices every 60s when online minimizes how stale data gets

### 4.2 Refund Processing Offline

If a customer returns an item in-store while the POS is offline, the refund is recorded locally. On sync:
- The server must verify the original order exists and the item was indeed sold
- Edge case: what if the order being refunded was itself created on another offline device and hasn't synced yet?
- **Mitigation:** local IndexedDB stores a cache of recent transactions for this store, so the POS can at least verify against local data. If the original order isn't found locally, the staff can still process the return but it gets flagged `needs_verification` on sync.

### 4.3 Device Registration & Authentication

No mention of how POS devices are authenticated. Each device needs:

```
pos_devices (
  id, store_id, device_name, device_uuid,
  api_key_hash, is_active,
  last_synced_at, last_ip,
  registered_by_admin_id, registered_at
)
```

- Device authenticates with a long-lived API key (stored in the PWA's IndexedDB, not in localStorage which is ephemeral)
- Super admin can revoke a device's access from the admin panel
- Every sync request includes `device_id` so the server can track per-device sync status

### 4.4 IndexedDB Data Volume

The plan scopes data to the store's own catalog. But even a single store with 5,000 variants × 4 sizes × 6 colors = 120,000 variant rows, plus prices, stock counts, and recent transaction history. Define explicit data retention:

```
Local cache:
  - Product catalog: full sync, refreshed every 60s when online
  - Inventory summary: full sync, refreshed every 60s
  - Recent transactions: last 30 days only
  - Customer lookup: last 90 days + frequent buyers
  - Outbox: retain for 7 days after successful sync, then prune
```

### 4.5 Service Worker Lifecycle

The PWA's Service Worker must handle:
- `install`: pre-cache app shell
- `activate`: clean old caches
- `fetch`: serve from cache first for app shell, network-first for API calls, falling back to cached data
- `sync`: Background Sync API for outbox replay (with fallback to manual sync button if browser doesn't support Background Sync)
- `push`: for receiving real-time stock update notifications when online

Not all browsers (especially on the desktop POS machines common in Indian retail) support Background Sync API. **Must have a manual "Sync Now" button** as a fallback.

## 5. Security — Missing Items

| Concern | Current Status | Recommendation |
|---|---|---|
| Input validation | Not mentioned | Use Zod/Joi for every API input. Generate validation middleware from the same schemas. |
| SQL injection | Not mentioned | Use a query builder or ORM (Knex/TypeORM/Prisma) with parameterized queries. Ban raw string interpolation. |
| Rate limiting | Mentioned in §5.18 but no architecture | Redis-based sliding window: 100 req/min for general API, 5 req/min for OTP, 10 req/min for guest checkout, 20 req/min for contact form |
| CORS | Not mentioned | Whitelist only known frontend origins. POS origin may be `null` (file://) — handle explicitly |
| CSRF | Not mentioned | For cookie-based auth (refresh tokens): double-submit cookie pattern or SameSite=Strict |
| JWT refresh rotation | Not detailed | On refresh token use, invalidate the old one and issue a new pair. Detect reuse → revoke all user sessions |
| File upload security | Not mentioned | Product images, courier receipt uploads: validate MIME type server-side (not just extension), limit file size, scan for malware on upload, serve from a separate domain/CDN (not same origin as API) |
| PII encryption | Mentioned in §5.9 | Specify: AES-256-GCM for phone/email at rest in PostgreSQL using pgcrypto or application-level encryption. Search requires HMAC-based blind index alongside the encrypted column |
| Admin session management | Mentioned in §4.10 | Specify: refresh tokens stored in DB with `is_revoked` flag. Super admin force-logout = set `is_revoked = true` + delete corresponding Redis session key |

## 6. Missing Modules / Features

### 6.1 Size Guide (Critical for Online Clothes Shopping)

Returns due to wrong sizing are the #1 cost driver in online fashion. A size guide tied to category (with measurements in cm/inches and a "how to measure" visual) should be in Phase 2, not deferred.

### 6.2 Coupon/Discount Engine

Mentioned in §5.6 as an edge case but not scoped into any phase. This is a revenue-driving feature for a fashion business (festive sales, clearance, first-order discounts). Recommend including in Phase 3 alongside rentals — rental pricing + coupon logic are both "pricing rule" problems.

### 6.3 Inventory Reconciliation (Stock Audit)

Physical stock will diverge from system stock over time (theft, miscounts, damage not logged). Need:

```
stock_audits (
  id, store_id, variant_id, system_qty, physical_qty,
  difference, reason, conducted_by_admin_id, conducted_at
)
```

A periodic stock audit workflow where staff count physical items, enter counts, and the system generates a variance report. Admin approves adjustments. This should be Phase 4 at latest.

### 6.4 Return & Exchange Policy Engine

The plan mentions "Returns/exchange workflow with reverse pickup scheduling" in §4.6 but doesn't define the rules. For a fashion business combining sales and rentals:

- **Sale returns:** within N days, tags attached, unused condition. Refund to original payment method. Store credit as alternative.
- **Rental late returns:** per-day penalty. Grace period?
- **Rental damage:** graded (minor/major/total loss). Each grade maps to a deduction percentage from deposit.
- **Exchange:** same item different size, subject to availability.

These rules must be configurable by admin, not hardcoded:

```
return_policies (
  id, type[sale_return/rental_late/rental_damage_minor/rental_damage_major/rental_loss],
  window_days, penalty_percent, is_active
)
```

### 6.5 Customer Loyalty / Store Credit

Even a simple store credit system (refund to wallet instead of original payment, with wallet balance usable on future purchases) significantly improves retention. Not in the plan at all. Consider:

```
wallet_transactions (
  id, user_id, amount, type[credit/debit],
  source[refund/promotion/cashback], reference_id, reference_type,
  balance_after, created_at
)
```

### 6.6 Product Sequencing / Merchandising

Admin needs control over which products appear first on the storefront (featured, new arrivals, trending). Add:

```
products (..., sort_priority INT, is_featured BOOLEAN, is_new BOOLEAN, tags TEXT[])
```

## 7. Timeline — Realism Check

The 16-week timeline is extremely optimistic for the stated scope, especially with the offline-first POS requirement. Here's a more realistic breakdown:

| Phase | Plan | Realistic | Why |
|---|---|---|---|
| Phase 1 — Foundation | 3 weeks | 4-5 weeks | Auth + RBAC with dynamic roles + permissions matrix + audit logs is 2 weeks alone. Product catalog with variants + images + categories is another 2. |
| Phase 2 — Core Commerce | 4 weeks | 7-8 weeks | This phase packs in cart, guest checkout, Razorpay, orders, POS with offline sync + IndexedDB + conflict resolution, lock mechanism, manual courier, GST invoice. The POS offline architecture alone is 3-4 weeks of focused work. |
| Phase 3 — Rentals | 3 weeks | 4-5 weeks | Rental calendar, deposit handling (online + offline), return/inspection/damage workflow, late fees, supplementary invoices. |
| Phase 4 — Engagement & Insights | 3 weeks | 3-4 weeks | Wishlist, reviews, reminders (BullMQ delayed jobs), dashboard. Most straightforward phase. |
| Phase 5 — Reporting & Hardening | 3 weeks | 4-5 weeks | PDF/Excel exports, contact/inquiry, performance tuning, security review, load testing, K8s deployment + monitoring setup. |
| **Total** | **16 weeks** | **22-27 weeks** | |

**Key risk:** Phase 2 is the danger zone. Trying to build offline-first POS + sync engine + GST invoicing + Razorpay in 4 weeks will result in either cut corners or missed deadlines. Recommend splitting Phase 2:

- **Phase 2a (4 weeks):** Cart, checkout, Razorpay, orders (online only), GST invoices
- **Phase 2b (3-4 weeks):** POS UI, IndexedDB layer, sync engine, lock mechanism, offline conflict resolution

This also means Phase 2a delivers a working online store earlier — useful for stakeholder demos and early testing.

## 8. Additional Recommendations

### 8.1 API Documentation

Add OpenAPI/Swagger spec generation from day one. Auto-generate from route definitions (using `tsoa` if TypeScript, or `swagger-jsdoc`). The POS sync API contract is especially critical — the offline client team needs a stable, versioned spec to build against.

### 8.2 Testing Strategy

Not mentioned in the plan at all. Recommend:

```
Unit tests:       Business logic (pricing, rental availability, stock deduction)
Integration tests: API endpoints against a test PostgreSQL + Redis
E2E tests:        Critical paths (checkout, rental booking, offline sync)
Contract tests:    POS sync API (verify client and server agree on shapes)
Load tests:       k6 or Artillery — simulate 100 concurrent checkouts hitting the same SKU
```

Priority test scenarios:
1. Two users checking out the last unit simultaneously
2. POS device selling offline, syncing after online sale of same unit
3. Rental booking for dates that overlap with an existing booking
4. Razorpay webhook arriving twice (idempotency)
5. Invoice numbering under concurrent order creation

### 8.3 Feature Flags

For a system this complex, deploy incomplete features behind feature flags (e.g., `RENTAL_ENABLED=false` until Phase 3 is ready). Use a simple DB-backed flag table or environment variables — no need for LaunchDarkly at this scale.

```
feature_flags (key, is_enabled, config_json, updated_by_admin_id, updated_at)
```

### 8.4 GraphQL vs REST Decision

The diagram mentions "REST/GraphQL" but the plan never decides. **Recommendation:** Start with REST. GraphQL adds complexity (N+1 queries, permission scoping per field, caching) that isn't justified at this stage. The storefront has predictable data access patterns — REST with well-designed endpoints is simpler to build, debug, cache, and secure. If the admin dashboard needs flexible querying later, add GraphQL only for the admin API.

### 8.5 TypeORM / Prisma vs Knex

Given the complex queries needed (rental availability, inventory aggregation, invoice numbering with advisory locks), an ORM may get in the way. **Recommendation:** Prisma for CRUD-heavy modules (products, users, orders) with raw query escape hatches for complex queries, or Knex + a thin model layer for full control. Avoid TypeORM — it has well-documented issues with complex relations and migrations.

### 8.6 Frontend State Management

The plan mentions "Redux/Zustand" but doesn't decide. For this application:

- **Zustand** for global UI state (auth, cart badge, sync status)
- **React Query** for all server state (products, orders, inventory) — this eliminates 80% of what Redux would be used for
- **IndexedDB (via Dexie.js)** for POS offline data
- **No Redux** — it adds boilerplate without benefit over Zustand + React Query

## 9. Revised Schema — Comprehensive

Below is the full schema with all gaps filled. This is what I'd hand to a developer to start implementing:

```sql
-- CORE: Users & Auth
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL,
  email VARCHAR(255) UNIQUE,
  phone VARCHAR(20) UNIQUE,
  password_hash VARCHAR(255),
  is_guest BOOLEAN DEFAULT false,
  is_phone_verified BOOLEAN DEFAULT false,
  is_email_verified BOOLEAN DEFAULT false,
  is_active BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  deleted_at TIMESTAMPTZ
);

-- Phone/email encrypted versions for PII compliance
-- (application-level AES-256-GCM encryption + HMAC blind index for search)

CREATE TABLE user_addresses (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id),
  label VARCHAR(50),  -- Home, Work, Other
  line1 TEXT NOT NULL,
  line2 TEXT,
  city VARCHAR(100) NOT NULL,
  state VARCHAR(100) NOT NULL,
  pincode VARCHAR(10) NOT NULL,
  country VARCHAR(3) DEFAULT 'IND',
  is_default BOOLEAN DEFAULT false,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  deleted_at TIMESTAMPTZ
);
CREATE INDEX idx_user_addresses_user ON user_addresses(user_id) WHERE deleted_at IS NULL;

-- CORE: Admin & RBAC
CREATE TABLE store_locations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL,
  address TEXT NOT NULL,
  city VARCHAR(100),
  state VARCHAR(100) NOT NULL,
  pincode VARCHAR(10),
  gstin VARCHAR(20) NOT NULL,  -- GST Identification Number
  phone VARCHAR(20),
  is_active BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE admin_users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL,
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  role_id UUID NOT NULL REFERENCES roles(id),
  store_ids UUID[],  -- which stores this admin has access to
  is_active BOOLEAN DEFAULT true,
  last_login_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  deleted_at TIMESTAMPTZ
);

CREATE TABLE roles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(100) NOT NULL,
  description TEXT,
  created_by_admin_id UUID REFERENCES admin_users(id),
  is_system BOOLEAN DEFAULT false,  -- true for Super Admin role (non-deletable)
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE permissions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  module VARCHAR(100) NOT NULL,  -- 'products', 'orders', 'rentals', 'reports', etc.
  action VARCHAR(50) NOT NULL,   -- 'view', 'create', 'edit', 'delete', 'export'
  description TEXT,
  UNIQUE(module, action)
);

CREATE TABLE role_permissions (
  role_id UUID NOT NULL REFERENCES roles(id) ON DELETE CASCADE,
  permission_id UUID NOT NULL REFERENCES permissions(id) ON DELETE CASCADE,
  PRIMARY KEY (role_id, permission_id)
);

-- CORE: Catalog
CREATE TABLE categories (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL,
  slug VARCHAR(255) UNIQUE NOT NULL,
  parent_id UUID REFERENCES categories(id),
  description TEXT,
  image_url TEXT,
  sort_order INT DEFAULT 0,
  is_active BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_categories_parent ON categories(parent_id);
CREATE INDEX idx_categories_slug ON categories(slug);

CREATE TABLE brands (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL UNIQUE,
  description TEXT,
  is_active BOOLEAN DEFAULT true
);

CREATE TABLE products (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(500) NOT NULL,
  slug VARCHAR(500) UNIQUE NOT NULL,
  description TEXT,
  category_id UUID REFERENCES categories(id),
  brand_id UUID REFERENCES brands(id),
  fabric VARCHAR(100),
  hsn_code VARCHAR(20),
  is_rentable BOOLEAN DEFAULT false,
  is_sellable BOOLEAN DEFAULT true,
  care_instructions TEXT,
  sort_priority INT DEFAULT 0,
  is_featured BOOLEAN DEFAULT false,
  is_active BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  deleted_at TIMESTAMPTZ
);
CREATE INDEX idx_products_category ON products(category_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_products_brand ON products(brand_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_products_slug ON products(slug) WHERE deleted_at IS NULL;

CREATE TABLE product_variants (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  product_id UUID NOT NULL REFERENCES products(id) ON DELETE CASCADE,
  size VARCHAR(20) NOT NULL,
```

## 10. Summary of Recommendations

| # | Area | Recommendation | Priority |
|---|---|---|---|
| 1 | Schema | Add all missing tables from §9 above before Phase 1 coding starts | Critical |
| 2 | Invoice numbering | Use Postgres advisory locks, not app-side MAX+1 | Critical |
| 3 | Inventory model | Hybrid: `inventory_units` for rental items, `inventory_summary` for sale-only | Critical |
| 4 | Rental module | Separate `rental_bookings` from `order_items`; model full state machine | Critical |
| 5 | Timeline | Split Phase 2 into 2a (online) and 2b (POS); extend to 22-27 weeks | High |
| 6 | Offline POS | Add device registration, price-stale detection, refund-offline handling | High |
| 7 | Security | Add input validation, rate limiting, CORS, CSRF, JWT rotation specifics | High |
| 8 | Rental extensions | Add `rental_extensions` table and availability re-check flow | Medium |
| 9 | Deposit accounting | GST on forfeited deposits; separate payment types | Medium |
| 10 | E-invoicing | Add IRN columns now even if below threshold | Medium |
| 11 | Size guide | Add `size_guide` table; include in Phase 2 | Medium |
| 12 | Coupon engine | Include in Phase 3; add `coupons` and `coupon_usages` tables | Medium |
| 13 | Stock audit | Add `stock_audits` table; include in Phase 4 | Medium |
| 14 | Wallet/Store credit | Add `wallet_transactions` for refund-to-wallet flow | Low |
| 15 | Architecture diagram | Update to reflect modular monolith with future split notation | Low |
| 16 | API contract | Define OpenAPI spec, error envelope, versioning from day one | Medium |
| 17 | Testing | Define strategy before Phase 1; 5 critical test scenarios listed above | High |
| 18 | Feature flags | Add `feature_flags` table; use for rental, POS, coupon toggles | Medium |
| 19 | Frontend state | Decide: Zustand + React Query + Dexie.js (no Redux) | Low |
| 20 | ORM choice | Prisma for CRUD + raw queries for complex; avoid TypeORM | Low |

The plan is strong in its business requirements coverage and the offline-first POS architecture is well-thought-out. The gaps are primarily in schema completeness, rental operational detail, security specificity, and timeline realism — all fixable before development begins.
